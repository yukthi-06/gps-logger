# Implementation Plan - APK Size Reduction & Optimization

Our analysis shows that the `55MB` APK size is primarily driven by the **MapLibre SDK (`org.maplibre.gl:android-sdk`)**, which includes heavy, native C++ compilation binaries (`.so` files) for four distinct CPU architectures (armeabi-v7a, arm64-v8a, x86, and x86_64).

We will implement two major production-grade optimizations in `app/build.gradle` to compress and shrink the APK to its absolute minimum size.

## Solution Design

### 1. Enable R8 Minification, Obfuscation & Resource Shrinking
Currently, the `release` build type is configured with `minifyEnabled false` and no resource shrinking.
- We will enable `minifyEnabled true` to allow R8 to perform aggressive dead-code elimination, stripping out all unused classes, methods, and dependencies from the final compiled dex.
- We will enable `shrinkResources true` to analyze your resources (layouts, drawables) and dynamically purge any unused files from the final package.

### 2. Enable ABI Splitting (Architecture-Specific APKs)
Bundling native render engines for every architecture into a single "universal" APK forces a device to download 4x the required C++ code.
- We will configure **ABI Splits** in Gradle. This automatically instructs Gradle to build separate, optimized APKs for each architecture:
  - `armeabi-v7a` (older phones)
  - `arm64-v8a` (99% of modern physical phones)
  - `x86` / `x86_64` (emulator and dev devices)
- The resulting architecture-optimized APKs (like the one for `arm64-v8a`) will plummet from `55MB` down to approximately **`12MB - 15MB`**!
- We will keep `universalApk true` enabled so that a fallback all-in-one APK is still built if needed.

## Proposed Changes

### 1. Update `app/build.gradle`
- Add `splits` configuration inside `android { ... }` block.
- Update `release` build type:
  - Set `minifyEnabled true`.
  - Add `shrinkResources true`.
