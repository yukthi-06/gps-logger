# Implementation Plan - Track List Polish & Single Point Parse Bug Fix

This document records the modifications made to strip redundant labels from the track/point display name and resolve the MapLibre parsing error when loading single location points on the map.

## 1. Description Cleanup (`TrackAdapter.java`)
- Previously, single-point entries were displayed as `"Recorded Point [User Note]"` or `"Recorded Point"`.
- The display name rendering has been updated:
  - If it starts with `"Recorded Point "`, that prefix is cleanly removed, leaving only the user's specific note/description (e.g. `"Home"` instead of `"Recorded Point Home"`).
  - If no custom description was given (resulting in exactly `"Recorded Point"`), it displays a clean generic label `"Single Point"` (prefixed by `📍 `).

## 2. Map Parsing Fix (`MapActivity.java`)
- **Root Cause of "Failed to parse" Error**:
  - The MapLibre SDK's `LatLngBounds.Builder.build()` method throws an `InvalidLatLngBoundsException` if there are fewer than `2` coordinates in the builder.
  - Since single-point KML, GPX, and JSON files only have exactly `1` coordinate fix, `boundsBuilder.build()` crashed, triggering the catch block and showing the `"Failed to parse track data."` toast.
- **Resolution**:
  - Safeguarded `boundsBuilder` generation when `points.size() == 1`.
  - Added two mock boundary anchors offset by a tiny fractional offset (`0.00025` degrees, ~25 meters) to the builder.
  - This guarantees that `boundsBuilder.build()` succeeds perfectly, generating a valid `LatLngBounds` enclosing the single coordinate.
  - MapLibre's `ensureMinBoundsSize` and camera easer load the single coordinate seamlessly and center/zoom directly over it.
