# Implementation Plan - Fix Rename Timestamp and Prefix Removal

We will update the renaming logic to guarantee that the `gps_` / `location_logs_` prefix and original timestamp are perfectly preserved when renaming a track or point.

## Root Cause Analysis
- In the initial implementation of `createNewBaseName`, we extracted the timestamp by checking if the base name after the prefix matched a regex `\\d{4}-\\d{2}-\\d{2}[.T]\\d{6}` with exactly 17 characters (like `2026-05-17.120000`).
- However, the actual timestamps produced by `ZonedDateTime.now().format(DateTimeFormatter.ISO_LOCAL_DATE_TIME)` contain colons (like `2026-06-01T11:08:45`), which failed the `\\d{6}` (6 digits with no colons) check.
- Because it failed the regex check, the track item fell through to the custom name fallback (Step 3), returning just the sanitized description (e.g., `my_trip`), stripping the prefix and timestamp.

## Solution Design
We will replace the regex-based timestamp check with a robust, dynamic string splitting mechanism that mirrors how `parseTrackItemDetails` extracts metadata:
1. **Recorded Points**: Continue using `_Recorded_Point` as the delimiter to split the name. Everything before it is the prefix + timestamp.
2. **Tracks**:
   - Check if the name starts with `gps_` or `location_logs_`.
   - If it does, extract the prefix.
   - The timestamp is everything after the prefix up to the first underscore `_` (which acts as the delimiter before the human-readable description).
   - If there is no underscore, the entire remaining string is the timestamp.
   - The new name is formed by combining `[PREFIX][TIMESTAMP]` with `_` and the sanitized new description.
3. **Fallbacks**: If no prefix exists, it safely falls back to using the custom user name.

This ensures all standard and legacy filename patterns are perfectly preserved.
